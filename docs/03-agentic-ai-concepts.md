# 03 — Agentic AI Concepts

## What is an AI agent?

An **AI agent** is an AI model that doesn't just respond to a single prompt — it takes a sequence of actions, uses tools, observes results, and continues reasoning until it completes a goal.

The key difference from a standard AI chatbot: an agent has a **loop**.

```
Goal → Reason → Act → Observe → Reason → Act → Observe → ... → Done
```

At each step, the agent decides whether it has enough information to complete the goal, or whether it needs to take another action (call a tool, read a file, query an API) before it can proceed.

## The three things an agent needs

1. **A model** — the AI that does the reasoning. (e.g., Claude, GPT-4, Gemini)
2. **Tools** — capabilities the model can invoke. (e.g., Terraform MCP server, a file system, a search API)
3. **A goal** — the natural language task the agent is trying to complete.

## The reasoning loop, in detail

| Step | What happens |
|------|--------------|
| **Receive goal** | The agent gets a task in natural language: "spin up a staging environment" |
| **Plan** | The model decides what actions are needed and in what order |
| **Act** | The model calls a tool (e.g., `terraform_plan`) |
| **Observe** | The model reads the tool's output (the plan output, any errors) |
| **Re-plan** | Based on what it observed, the model decides whether to continue, adjust, or stop |
| **Complete** | The goal is met — the model returns a summary to the user |

## What makes this different from automation scripts

Traditional automation (scripts, CI/CD pipelines) follows a fixed path. It does exactly what it was programmed to do. If something unexpected happens, it fails or requires human intervention.

An AI agent can **adapt**. If a `terraform_plan` shows an error, the agent can reason about the error, attempt a fix, re-plan, and retry — without a human in the loop.

This is not magic. The agent is bounded by the tools it has access to and the quality of its reasoning. But for well-defined infrastructure workflows, it can handle a surprising amount of variation.

## A note on trust and control

Agentic systems require thoughtful guardrails. Key considerations for customer conversations:

- **Human-in-the-loop checkpoints** — many production deployments require a human to approve `terraform_apply` even if the plan was generated by an agent.
- **Scope limiting** — agents can be restricted to specific workspaces, regions, or resource types.
- **Dry-run modes** — agents can be configured to plan but never apply without explicit approval.

These aren't weaknesses. They're the right design for most enterprise environments.
